# LockIn Bot - Agent Guidelines

This file contains essential information for agentic coding agents working on this Discord bot project.

## Project Overview

LockIn Bot is a Go-based Discord bot that gamifies studying through voice channel tracking, streak systems, and leaderboards. It uses Clean Architecture with PostgreSQL and the discordgo library.

**Stack**: Go 1.24.1, PostgreSQL 17, Discord API (discordgo), SQLC for type-safe queries

## Build & Test Commands

### Development
```bash
# Build the bot
go build -o lockin-bot ./main.go

# Run the bot
./lockin-bot
# OR
go run main.go

# Run all tests
go test ./...

# Run tests with verbose output
go test ./... -v

# Run a specific test
go test ./internal/bot -run TestHandleSlashStatsCommand_UserNotFound_CreatesUser -v

# Run tests for specific package
go test ./internal/service -v
```

### Database & Migrations
```bash
# Install goose (if needed)
go install github.com/pressly/goose/v3/cmd/goose@latest

# Run migrations
goose -dir db/migrations postgres "your_database_url" up

# Generate SQLC queries (after modifying db/queries.sql)
sqlc generate
```

### Docker
```bash
docker build -t lockin-bot .
docker run --env-file .env lockin-bot
```

## Code Style Guidelines

### Imports
- **Ordering**: Standard library → internal imports → external dependencies
- **Groups**: Separate groups with blank line
- **Alphabetical**: Within each group, sort alphabetically
```go
import (
    "context"
    "log"

    "github.com/Skufu/LockIn-Bot/internal/config"
    "github.com/Skufu/LockIn-Bot/internal/database"
    "github.com/bwmarrin/discordgo"
)
```

### Naming Conventions
- **Types/Interfaces**: PascalCase (e.g., `Bot`, `StreakService`, `MockQuerier`)
- **Exported functions**: PascalCase (e.g., `New`, `Connect`)
- **Private functions**: camelCase (e.g., `handleVoiceStateUpdate`)
- **Constants**: UPPERCASE_SNAKE_CASE (e.g., `minimumActivityMinutes`)
- **Interface methods**: PascalCase

### Comments
- Standard Go conventions: `// Comment starts with capital letter`
- Package comments on line 1: `// Package bot provides Discord bot functionality`
- Exported comments before declarations
- Inline comments explain "why", not "what"

### Error Handling
```go
// Check errors explicitly and return to callers
result, err := db.GetUser(ctx, userID)
if err != nil {
    return nil, fmt.Errorf("failed to get user: %w", err)
}

// Log non-fatal errors with context
if err != nil {
    log.Printf("Warning: Failed to update stats for user %s: %v", userID, err)
}

// Fatal errors only at startup
if err != nil {
    log.Fatalf("Failed to connect to database: %v", err)
}
```

### Concurrency
- Use `sync.Mutex` for shared state protection
- Use worker pools (`chan func()`) to prevent goroutine explosion
- Always use `defer` when unlocking mutexes
- Background tasks in goroutines with shutdown channels

### Database & SQLC
- All database queries generated by SQLC in `internal/database/queries.sql.go`
- Use type-safe queries via `db.Querier`
- Nullable fields use `sql.NullString`, `sql.NullTime`, `sql.NullInt32`
- Run `sqlc generate` after modifying `db/queries.sql`

### Timezone Handling
**CRITICAL**: All timezone-sensitive operations use Manila timezone
- Use helper functions from `internal/service/timezone_helpers.go`
- All streak calculations based on Manila calendar days
- Consistent timezone prevents gaming and ensures fairness

### Clean Architecture Layers

1. **External Layer** (`internal/bot/`): Discord event handlers, session management
2. **Application Layer** (`internal/commands/`): Command routing, input validation
3. **Business Layer** (`internal/service/`): Core logic (streaks, statistics)
4. **Data Layer** (`internal/database/`): PostgreSQL operations, SQLC queries

**Dependency Flow**: External → Application → Business → Data (never reverse)

### Configuration
- Environment variables via `.env` file (use `.env.example` as template)
- Load with `config.Load()` in `internal/config/config.go`
- Validate required variables (DISCORD_TOKEN, DB_PASSWORD)
- Optional variables log warnings if missing

### Discord Integration
- Use `discordgo` library
- Register handlers with `session.AddHandler()`
- Voice tracking: check against `AllowedVoiceChannelIDsMap`
- Slash commands: Register via `session.ApplicationCommandCreate()`

## Common Patterns

### Creating a new service
```go
type NewService struct {
    db  *database.Queries
    cfg *config.Config
}

func NewNewService(queries *database.Queries, appConfig *config.Config) *NewService {
    return &NewService{
        db:  queries,
        cfg: appConfig,
    }
}
```

### Handling nullable database values
```go
if stats.TotalStudyMs.Valid {
    totalMs = stats.TotalStudyMs.Int64
}
```

### Context-aware operations
```go
func HandleSomething(ctx context.Context, userID string) error {
    result, err := s.db.GetUser(ctx, userID)
    if err != nil {
        return fmt.Errorf("failed to get user: %w", err)
    }
    // ...
}
```

## Testing Requirements

- Write tests for new features
- Use mock implementations for database operations
- Test should call the actual code path, not just database operations
- Test both success and error cases
- Use table-driven tests for multiple scenarios

## Verification Before Pushing

- [ ] Run `go test ./...` - all tests pass
- [ ] Run `go build ./...` - no build errors
- [ ] Run `sqlc generate` if you modified SQL queries
- [ ] Ensure timezone-sensitive code uses Manila helpers
- [ ] Check nil pointer safety with nullable fields
- [ ] Verify error handling doesn't panic
